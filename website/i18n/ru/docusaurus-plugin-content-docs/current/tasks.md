# Задачи

У задачи есть имя и тело. Две формы записи:

```lua
-- простая функция: без зависимостей
task("hello", function()
    print("привет")
end)

-- таблица-спецификация
task("build", {
    desc = "собрать проект",        -- необязательно; видно в `taku list`
    deps = { "gen", "fmt" },        -- необязательно; выполняются первыми
    run = function() ... end,       -- обязательно
})
```

## Зависимости

`taku run <task>` сначала выполняет транзитивные зависимости. Каждая задача
выполняется **ровно один раз за запуск**, даже если её указали в `deps` несколько
задач — общая зависимость никогда не повторяется. Неизвестные задачи и циклы
зависимостей сообщаются как ошибки до начала выполнения.

```lua
task("gen",   { run = function() ... end })
task("build", { deps = { "gen" }, run = function() ... end })
task("test",  { deps = { "gen" }, run = function() ... end })
task("ci",    { deps = { "build", "test" } })   -- `gen` всё равно выполнится один раз
```

## Параллелизм

Независимые задачи выполняются одновременно в отдельных потоках — по умолчанию
вплоть до числа ядер CPU; ограничить можно через `taku run <task> -j N`. Первая
ошибка останавливает запуск новых задач; уже запущенные завершаются.

Тело каждой задачи исполняется в **своей свежей песочнице** — общего изменяемого
Lua-состояния между задачами нет, а код верхнего уровня Takefile выполняется по
разу на каждую запускаемую задачу. Держите на верхнем уровне только определения
задач.

## Импорты

Большой Takefile можно разбить на несколько файлов через `import`:

```lua
import("tasks/build.lua")
import("tasks/deploy.lua")
```

- **Относительные пути.** Путь разрешается относительно каталога файла, который
  вызывает `import`, а не относительно текущего рабочего каталога. Вложенные
  импорты разрешаются относительно своего файла, поэтому `tasks/build.lua` может
  вызвать `import("helpers.lua")`, чтобы подключить `tasks/helpers.lua`.
- **Только `.lua`, та же песочница.** `import` читает текстовый `.lua`-файл и
  выполняет его в том же ограниченном состоянии — импортированный код получает те
  же глобалы `fs`/`sh`/... и те же [ограничения](./sandbox.md), поэтому выйти из
  песочницы не может. Отсутствующий файл вызывает ошибку.
- **Выполняется при загрузке, один раз.** Файл исполняется сразу в точке `import`;
  его вызовы `task(...)` верхнего уровня регистрируются в общий набор задач. Каждый
  файл импортируется **не более одного раза** — повторный импорт, даже по другому
  пути к тому же файлу, ничего не делает. Поэтому два файла могут безопасно
  `import` один общий helper.
- **Только побочные эффекты.** `import` ничего не возвращает; используйте его,
  чтобы зарегистрировать задачи или определить общие глобалы, а не для возврата
  значения.
