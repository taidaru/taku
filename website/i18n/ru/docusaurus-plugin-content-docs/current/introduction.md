# Введение

Taku — таск-раннер. Вы описываете задачи проекта — сборку, тесты, деплой — в
файле `Takufile.lua`, а Taku выполняет их вместе с зависимостями, по
возможности параллельно.

Ключевая идея: **таск — это план, а не скрипт**. Тело таска — список шагов,
который Lua строит при загрузке файла, а исполняет ядро на Rust. Поскольку
план — это данные, Taku может его напечатать (`--dry-run`), пропустить, если
входы не изменились (`unchanged`), и объяснить свои решения (`--explain`) —
то, чего не умеет shell-скрипт.

```lua
--- сгенерировать исходники
task "gen" {
    write { "1.0.0", to = "version.txt" },
}

--- собрать проект
task "build <profile=dev>: gen" {
    unchanged { "src/**/*.rs", outputs = "target" },
    "cargo build --profile ${profile}",
}
```

```sh
taku run build                          # gen, затем build
taku run build                          # пропуск: ничего не изменилось
taku run build --vars profile=release   # переопределение параметра
taku run build --dry-run                # напечатать план, ничего не трогая
```

Что вы получаете:

- **Один файл, один язык.** Таски — это Lua: циклы, условия и работа со
  строками вместо синтаксиса Make или YAML.
- **Кроссплатформенность.** Команды выполняются без шелла; один и тот же
  Takufile работает на Linux, macOS и Windows.
- **Песочница.** `io`/`os` в Lua отключены. Takufile добирается до системы
  только через шаги и модули Taku, и только во время исполнения таска —
  загрузка Takufile никогда не имеет побочных эффектов. См.
  [Песочницу](./sandbox.md).
- **Инкрементальность.** Guard `unchanged` снимает фингерпринт входов и
  пропускает работу.
- **Сервисы.** `serve` запускает dev-серверы и базы, ждёт их готовности и
  гасит по завершении запуска.
