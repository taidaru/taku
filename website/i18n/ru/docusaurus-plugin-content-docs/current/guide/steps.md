# Шаги

Шаг — это **данные**: голый глагол вроде `rm "path"` или `write {...}` сам по
себе ничего не делает — он возвращает описание, которое раннер исполняет по
порядку. Именно это делает возможными `--dry-run`, `unchanged` и читаемый
вывод ошибок.

Все конструкторы шагов:

| Шаг | Действие |
|---|---|
| `"cargo build"` | выполнить команду |
| `argv { "prog", "arg", ... }` | команда из явного списка аргументов |
| `pipe { "cmd1", "cmd2" }` | конвейер |
| `rm "path"` | удалить файл или каталог (рекурсивно) |
| `mkdir "path"` | создать каталог с родителями |
| `cp { "src", to = "dst" }` | скопировать файл |
| `mv { "src", to = "dst" }` | переместить / переименовать |
| `write { "data", to = "file" }` | записать файл |
| `append { "line", to = "file" }` | дописать строку (с переводом строки) |
| `download { "url", to = "file", sha256 = "..." }` | скачать по HTTP(S) |
| `echo "text"` | напечатать строку |
| `confirm "вопрос?"` | спросить `[y/N]`; отказ роняет таск |
| `invoke "task"` | выполнить шаги другого таска на месте |
| `unchanged { ... }` | пропустить остаток, если входы не менялись — [Инкрементальность](./incremental.md) |
| `serve { ... }` | запустить долгоживущий сервис — [Сервисы](./services.md) |
| `function(ctx) ... end` | произвольный Lua — [Переменные](./variables.md) |

Каждый строковый аргумент — шаблон: плейсхолдеры `${парам}` и `$ENV_VAR`
разрешаются непосредственно перед исполнением шага (см.
[Переменные](./variables.md)).

## Команды

Голая строка выполняет команду. Строка токенизируется так, как её прочитал бы
шелл — кавычки группируют слова, — но **шелл не участвует**: ничего не
раскрывается и не переинтерпретируется, интерполированные значения не могут
внедрить лишние команды.

```lua
task "build <profile=dev>" {
    "cargo build --profile ${profile}",
    "sh -c 'grep TODO src/*.rs | wc -l'",   -- нужен шелл? запустите его явно
}
```

Ненулевой код завершения роняет шаг и таск. Сообщение об ошибке показывает
**шаблон** команды, а не разрешённые значения — секреты из окружения не
попадают в вывод.

Опции задаются табличной формой, команда — на позиции 1:

```lua
{ "make install", cwd = "sub", env = { CC = "clang" }, allow_fail = true, timeout = 60 }
```

| Опция | Значение |
|---|---|
| `cwd` | рабочий каталог |
| `env = {...}` | дополнительные переменные окружения для этой команды |
| `allow_fail = true` | ненулевой код не роняет таск |
| `timeout` | секунды; по истечении процесс убивается и шаг падает |
| `stdin` | строка, подаваемая на stdin команды |

Процессы команд наследуют окружение Taku; файл [`.env`](../api/env.md)
дозаполняет переменные, не заданные окружением, а `env =` шага побеждает и
то и другое.

## argv и pipe

`argv` принимает список аргументов явно. Каждый элемент форматируется отдельно
и никогда не перетокенизируется — используйте, когда значение может содержать
пробелы или кавычки:

```lua
argv { "git", "commit", "-m", "${message}" }
```

`pipe` соединяет команды как `a | b | c`, средствами раннера. Падение любой
стадии роняет шаг (pipefail — всегда):

```lua
pipe { "cat access.log", "grep ' 500 '", "wc -l" }
```

Оба принимают те же опции, что и команды.

## Файлы

`rm` и `mkdir` принимают путь; `cp`, `mv`, `write`, `append` — таблицу с
данными на позиции 1 и целью в `to`:

```lua
task "dist" {
    mkdir "dist",
    cp { "target/release/app", to = "dist/app-${version}" },
    write { "${version}", to = "dist/VERSION" },
    append { "built ${version}", to = "build.log" },
    rm "tmp",
}
```

## download

```lua
download {
    "https://example.com/tool.tar.gz",
    to = "vendor/tool.tar.gz",
    sha256 = "9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08",
}
```

Если задан `sha256`, несовпадение хэша удаляет скачанный файл и роняет шаг.

## confirm

`confirm "вопрос?"` печатает вопрос и ждёт `y`. Всё остальное роняет таск.
Под `--yes` или без терминала (CI) отвечает сам:

```lua
task "db-reset" {
    confirm "стереть локальную базу?",
    "dropdb app && createdb app",
}
```

## invoke

`invoke "имя"` выполняет шаги другого таска прямо здесь, внутри текущего — в
отличие от зависимости, которая выполняется до старта таска.
`invoke "имя" { p = v }` передаёт значения параметров.

Invoke выполняется всегда. Но его завершение засчитывается: если позже
какой-то таск зависит от вызванного, эта зависимость уже удовлетворена.
